🧪 Laboratorio: Monitorización de Pipelines DataStage con WatsonX Orchestrate y MCP Server

Objetivo

Crear un agente en IBM WatsonX Orchestrate que responda a preguntas de usuarios sobre la ejecución de pipelines de DataStage, consultando un MCP Server que accede a un fichero CSV con los datos de ejecución. El agente debe interpretar dependencias entre pipelines y regiones, usando un documento de conocimiento en Markdown.

🗂 Estructura del laboratorio

  • knowledge_document.md: Documento de conocimiento con regiones, pipelines y dependencias.
  • ejecuciones_pipelines.csv: Fichero con los datos de ejecución de pipelines.
  • mcp_server.py: Servidor MCP con endpoints REST para consulta de ejecuciones.
  • requirements.txt: Dependencias Python (Flask, pandas).

📊 Estados de Ejecución (run_status)

  • 0 (Fallido): El paso o el trabajo terminó con error.
  • 1 (Exitoso): El paso o el trabajo se completó correctamente.
  • 2 (Reintento): El paso falló, pero está configurado para reintentar.
  • 3 (Cancelado): El trabajo fue cancelado manualmente o por alguna condición.
  • 4 (En progreso): El paso está actualmente en ejecución.

🧱 Componentes

🐳 MCP Server

El MCP Server expone tres endpoints REST y consulta el fichero ejecuciones_pipelines.csv. No implementa lógica de dependencias; esta lógica la gestiona el agente en WatsonX Orchestrate usando el documento de conocimiento.

📁 Archivo de entrada

  • ejecuciones_pipelines.csv
  • Columnas: region, pipeline_name, start_time, end_time, run_status

Endpoints REST

1. /estado-ejecucion

Consulta el estado de ejecución de una pipeline en una región y día específico.

  • Parámetros de entrada: region, pipeline, dia
  • Respuesta: Region, Pipeline, Dia, Estado de ejecución, Log
2. /estadisticas-ejecucion

Devuelve estadísticas de ejecución en un intervalo de fechas.

  • Parámetros de entrada: region, pipeline, dia_inicio, dia_fin
  • Respuesta: Estadísticas generales y por estado (duración, horarios, etc.)
3. /nivel-actualizacion

Verifica si las dependencias de una pipeline se cumplieron correctamente ese día.

  • Parámetros de entrada: region, pipeline, dependencias_pipeline, dia

  • Respuesta: Para cada dependencia, indica si se cumplió, falló, no finalizó o fue fuera de orden.

    ** Situaciones de Salida:**

    0 (Proceso previo fallo) La ejecución del pipeline del que depende fallo ese día (run_status del pipeline del que depende fue igual a 0).

    1 (Dependencia satisfactoria): La ejecución del pipeline del que depende acabo satisfactoriamente (run_status del pipeline del que depende fue igual a 1) y la fecha de fin del pipeline del que depende es anterior o igual a la fecha de inicio del pipeline dependiente.

    2 (Dependencia no finalizo o fue cancelada): La ejecución del pipeline del que depende no finalizó ese día o fue cancelado (run_status del pipeline del que depende fue igual a 2, 3 o 4).

    3 (Proceso previo finalizó bien pero no en tiempo): La ejecución del pipeline del que depende acabo satisfactoriamente (run_status del pipeline del que depende fue igual a 1) y la fecha de fin del pipeline del que depende es posterior a la fecha de inicio del pipeline dependiente.

🚀 Instalación y Ejecución

🐳 Instala dependencias:

`bash pip install -r requirements.txt

🐳 Construcción de la imagen

`bash docker build -t mcp-server .

🐳 Ejecutar la imagen

`bash docker run -d -p 5000:5000 --name mcp-server mcp-server:latest

2. Agente en WatsonX Orchestrate

El agente: - Recibe preguntas de usuarios sobre la ejecución de pipelines - Consulta el MCP Server vía REST - Usa un documento de conocimiento en formato Markdown para entender: - Regiones disponibles - Pipelines disponibles - Dependencias entre pipelines
